# Заметки по проекту

## Процесс работы

- Для проверок запускать только `npm run check-and-fix`. Отдельно `npm run ts-check` не запускать.
- По возможности держать модули небольшими и с конкретной зоной ответственности, не смешивать всё в одном.
- Не оставлять в коде мёртвые или неиспользуемые куски кода без явного запроса или комментария в коде.
- Не возвращаться с вопросами к решениям, уже принятым в этой ветке, если только новая правка действительно не создаёт новый компромисс.

## Лейаут текста на карточке

- Логика подбора размера шрифта чувствительна к регрессиям; менять её аккуратно.
- Целевое поведение: каждый текстовый блок должен рендериться максимально возможным размером шрифта, помещаясь в контейнер шириной `85%` от карточки и максимум в `2` строки.
- Измерение текста должно быть стабильным во время взаимодействия. Оно не должно перезапускаться из-за drag, reveal, swipe или flip.
- Измерения должны опираться на фиксированный набор строк для конкретной стороны/языка карточки, а не на временное состояние интерфейса вроде текущего показанного варианта.
- Высоту блока вариантов нужно резервировать по самому высокому подходящему варианту на этой стороне/языке, чтобы центральный блок не прыгал при смене вариантов.
- Базовая линия текста должна оставаться визуально стабильной даже при смене размера шрифта.
- `line-height` для авторазмерных фразовых блоков задаётся от максимального размера текста. Логика измерений должна учитывать реальный `computed line-height`.
- Поведение с `NBSP` важно. И интерфейсные строки, и текст карточек используют неразрывные пробелы, чтобы короткие слова и предлоги не отрывались от следующего слова.

## Адаптивный лейаут

- При любых правках карточки, текста или верхнего summary всегда проверять и десктоп, и мобилу.
- На мобиле и десктопе должен сохраняться один и тот же визуальный формат карточки. Мобила — это масштабированная версия десктопной карточки, а не другой аспект.
- Для этого проекта типичны регрессии размера карточки: нельзя допускать состояния, где карточка схлопывается в полоску или становится слишком широкой на мобиле.
- Текст summary над колодой не должен ломать пропорции карточки. Если места мало, должен адаптироваться лейаут, а не искажаться карточка.
- При правках stage/deck-лейаута нужно сохранять достаточно места и для полного summary, и для полной карточки на мобиле и на десктопе.

## Правила интеракции

- Сторона карточки определяется один раз при старте для каждой карточки и дальше меняется только явным flip.
- Когда карточка выходит из колоды и становится основной, она должна сохранять сторону, с которой лежала в колоде.
- Фоновая колода должна вести себя визуально стабильно во время drag; нельзя заново вносить мигание из-за лишних ререндеров.

## Финальный экран

- При ошибке основная кнопка `Проверить` остаётся disabled, а под текстом ошибки показывается отдельная кнопка повтора.
- При успехе кнопка `Проверить` снова становится активной и может повторно запускать цикл `loading -> success -> animation`.
- Тексты успеха и ошибки берутся из `src/data/uiText.ts`; интерфейсные строки должны оставаться централизованными там и нормализоваться через `makeNBSP`.

## Тестовые хелперы

- Стрелки клавиатуры подставляют соответствующий вариант, но не выполняют свайп.
- Параметр `?bubu=true|false` предзаполняет стейт до последнего шага для проверки.

## Данные

- Русский и сербский контент должны быть синхронизированы по смыслу, если только нет намеренной языковой адаптации или шутки.
- Если меняются фрагменты фраз в данных колоды, нужно перепроверять, что правильная последовательность, скрытая пунктуация и итоговая собранная фраза по-прежнему согласованы между собой.
